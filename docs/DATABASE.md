# Luna Desktop AI Assistant — Database Design

> **Version:** 1.0 · July 2026
> **Engine:** SQLite 3 (file: `data/luna.db`)
> **ORM:** SQLAlchemy 2.0 with declarative base
> **Migrations:** Alembic
> Reference: [README.md](./README.md) · [API_SPEC.md](./API_SPEC.md)

---

## Overview

Luna uses two storage systems:

| System | Purpose | Location |
|--------|---------|---------|
| **SQLite** | All structured relational data | `data/luna.db` |
| **FAISS** | Vector indexes for semantic similarity search | `data/faiss/` |

SQLite is the source of truth for all data. FAISS stores float32 vectors and references back to SQLite rows via ID. If FAISS is lost, it can be rebuilt by re-embedding all rows in `memories` and `uploaded_files`.

## SQLite Schema

### Table: `conversations`

Stores chat session metadata.

```sql
CREATE TABLE conversations (
    id          TEXT PRIMARY KEY,           -- UUID v4
    title       TEXT NOT NULL DEFAULT 'New Conversation',
    created_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    is_deleted  INTEGER NOT NULL DEFAULT 0  -- soft delete flag
);

CREATE INDEX idx_conversations_updated_at ON conversations(updated_at DESC);
```

| Column | Type | Notes |
|--------|------|-------|
| `id` | TEXT (UUID) | Primary key, generated by application |
| `title` | TEXT | Editable by user, auto-generated from first message |
| `created_at` | DATETIME | UTC |
| `updated_at` | DATETIME | Updated on every new message |
| `is_deleted` | INTEGER | 0=active, 1=soft-deleted |

---

### Table: `messages`

Stores individual messages within conversations.

```sql
CREATE TABLE messages (
    id              TEXT PRIMARY KEY,
    conversation_id TEXT NOT NULL REFERENCES conversations(id) ON DELETE CASCADE,
    role            TEXT NOT NULL CHECK(role IN ('user', 'assistant', 'system')),
    content         TEXT NOT NULL,
    model_used      TEXT,                   -- e.g. 'qwen2.5:7b', null for user messages
    tokens_used     INTEGER,
    created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    metadata        TEXT                    -- JSON blob for extra data (attachments, action refs)
);

CREATE INDEX idx_messages_conversation_id ON messages(conversation_id);
CREATE INDEX idx_messages_created_at ON messages(created_at DESC);
```

| Column | Type | Notes |
|--------|------|-------|
| `id` | TEXT (UUID) | |
| `conversation_id` | TEXT (FK) | Cascades on delete |
| `role` | TEXT | `user`, `assistant`, or `system` |
| `content` | TEXT | Full message text |
| `model_used` | TEXT | Null for user messages |
| `tokens_used` | INTEGER | Approximate token count |
| `metadata` | TEXT | JSON: `{"attachments": [], "action_log_id": 42}` |

### Table: `memories`

Stores long-term memory entries extracted from conversations.

```sql
CREATE TABLE memories (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    content         TEXT NOT NULL,
    source          TEXT NOT NULL CHECK(source IN ('conversation', 'manual', 'file')),
    source_id       TEXT,                   -- conversation_id or file_id that generated this
    importance      REAL NOT NULL DEFAULT 0.5,  -- 0.0 to 1.0
    faiss_id        INTEGER,                -- ID in FAISS index (null if not yet indexed)
    created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    last_accessed   DATETIME,
    access_count    INTEGER NOT NULL DEFAULT 0,
    is_deleted      INTEGER NOT NULL DEFAULT 0
);

CREATE INDEX idx_memories_importance ON memories(importance DESC);
CREATE INDEX idx_memories_created_at ON memories(created_at DESC);
CREATE INDEX idx_memories_faiss_id ON memories(faiss_id);
```

| Column | Type | Notes |
|--------|------|-------|
| `id` | INTEGER | Auto-increment PK |
| `content` | TEXT | Raw memory text |
| `source` | TEXT | Where this memory came from |
| `importance` | REAL | 0.0 (low) to 1.0 (high) — affects retrieval ranking |
| `faiss_id` | INTEGER | Maps to FAISS vector index position |
| `access_count` | INTEGER | Incremented each time this memory is retrieved |

---

### Table: `embeddings_meta`

Maps FAISS vector IDs to their originating record. Needed because FAISS does not store metadata.

```sql
CREATE TABLE embeddings_meta (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    faiss_id    INTEGER NOT NULL UNIQUE,
    record_type TEXT NOT NULL CHECK(record_type IN ('memory', 'document_chunk')),
    record_id   TEXT NOT NULL,              -- memory.id or document_chunk reference
    index_name  TEXT NOT NULL,             -- 'memory' or 'docs'
    created_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_embeddings_faiss_id ON embeddings_meta(faiss_id);
CREATE INDEX idx_embeddings_record ON embeddings_meta(record_type, record_id);
```

### Table: `action_logs`

Immutable audit trail of all automation actions.

```sql
CREATE TABLE action_logs (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    action          TEXT NOT NULL,
    parameters      TEXT NOT NULL,          -- JSON blob
    result          TEXT,                   -- Result summary or error
    status          TEXT NOT NULL CHECK(status IN ('executed', 'denied', 'failed', 'timeout')),
    permission_type TEXT CHECK(permission_type IN ('pre_approved', 'allow_once', 'allow_always', 'denied')),
    user_approved   INTEGER,               -- 1=yes, 0=no, null=pre-approved
    risk_level      TEXT CHECK(risk_level IN ('low', 'medium', 'high')),
    conversation_id TEXT REFERENCES conversations(id),
    duration_ms     INTEGER,               -- Execution time in milliseconds
    created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_action_logs_created_at ON action_logs(created_at DESC);
CREATE INDEX idx_action_logs_action ON action_logs(action);
CREATE INDEX idx_action_logs_status ON action_logs(status);
```

### Table: `uploaded_files`

Metadata for files processed by the RAG pipeline.

```sql
CREATE TABLE uploaded_files (
    id              TEXT PRIMARY KEY,       -- UUID v4
    filename        TEXT NOT NULL,
    original_path   TEXT,
    mime_type       TEXT,
    file_size_bytes INTEGER,
    chunk_count     INTEGER,
    status          TEXT NOT NULL CHECK(status IN ('pending', 'indexing', 'indexed', 'failed')),
    conversation_id TEXT REFERENCES conversations(id),
    created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_uploaded_files_conversation ON uploaded_files(conversation_id);
CREATE INDEX idx_uploaded_files_status ON uploaded_files(status);
```

```
conversations ──< messages            (one-to-many, cascade delete)
```

## FAISS Index Design

Luna uses **two separate FAISS flat indexes**:

### 1. `data/faiss/memory.index`
- **Type:** `faiss.IndexFlatL2`
- **Dimension:** 384
- **Model:** BAAI/bge-small-en-v1.5
- **Content:** One vector per `memories` row
- **ID mapping:** `embeddings_meta` table (`index_name = 'memory'`)

### 2. `data/faiss/docs.index`
- **Type:** `faiss.IndexFlatL2`
- **Dimension:** 384
- **Content:** One vector per document chunk from `uploaded_files`
- **ID mapping:** `embeddings_meta` table (`index_name = 'docs'`)

**Persistence:**
- Both indexes are loaded into memory on backend startup.
- Both are saved to disk on every write and on backend shutdown.
- If index file does not exist, a new empty index is created.

**Rebuild Strategy:**
If FAISS index is corrupted or missing:
1. Create new empty `IndexFlatL2(384)`
2. Query all memories/chunks from SQLite where `faiss_id IS NOT NULL`
3. Re-embed each row and add to index
4. Update `faiss_id` in SQLite with new index positions
5. Save index to disk

## Migration Strategy

### Current: SQLite with Alembic

Alembic manages schema versions. Each schema change gets its own migration file in `alembic/versions/`.

**Migration workflow:**
```bash
alembic revision --autogenerate -m "description"
alembic upgrade head
```

**Rollback:**
```bash
alembic downgrade -1
```

### Future: PostgreSQL Migration Path

If multi-user or high-concurrency is needed:
1. Replace `sqlite:///data/luna.db` URL with `postgresql://...` in config
2. Run `alembic upgrade head` against PostgreSQL
3. Migrate data with pg_restore or custom ETL script
4. FAISS remains unchanged (filesystem-based)

All SQLAlchemy models are written to be DB-agnostic. No SQLite-specific SQL is used in ORM models.
